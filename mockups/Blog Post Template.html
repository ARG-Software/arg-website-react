How to actually enforce clean architecture in TypeScript

You wrote the docs. You did the PR reviews. And yet, six months later, controllers are directly importing repositories and nobody noticed.

MR
Mateo Ramírez
Staff Engineer, ARG
Nov 14, 2025 8 min read Architecture · TypeScript

You wrote the docs. You did the PR reviews. You explained it in the onboarding. And yet, six months later, a bunch of controllers are directly importing a repository, and nobody noticed.

The real problem with architecture

Clean Architecture, Hexagonal, Layered — whatever you're following, the rules only exist in two places: your head, and a README nobody reads.

The moment a deadline hits, someone takes a shortcut. A controller calls a repository directly. A domain model imports a NestJS decorator. An infrastructure class leaks into your application layer. It's not malicious — it's invisible. Nobody gets a red light. The tests still pass. The app still ships.

Six months later, you have a codebase that looks structured but behaves like a mess.

Architecture tests — rules that enforce themselves

The fix is to stop trusting humans to remember rules, and start writing rules that fail the build. Architecture tests run in your CI pipeline like any other test, but they assert on dependency graphs instead of behavior.

"If it can break silently, it will break silently. Move the check from the README to the build."

The contract is simple: domain must not import from infrastructure. application must not import from presentation. Anyone who violates the rule sees red before merge, not in production.

Asserting on dependency graphs

Instead of asserting on outputs, you assert on shape. Which modules can import which. Which symbols are allowed to cross a boundary. Once those facts are code, they live where every other invariant lives — in the test suite.

Writing your first contract

Start small. One rule, one layer. Pick the boundary you care about most and make a single test fail when it's crossed. Build from there.

The silent killer other libraries miss

Most tools check direct imports and stop there. The interesting violations are transitive: a use-case imports a helper that imports an ORM model. On paper the use-case is clean. In practice, your domain layer just got a database dependency.

Transitive imports and ORM leaks

A two-hop import is still an import. The fact that there's an innocent file in the middle doesn't change the dependency.

Walking the full graph

Walk the full graph, not just the first edge. The tools that get this right traverse every reachable module and ask the same question at every hop.

Why this changes everything

Once architecture is enforced by the build, the conversation shifts. Code review stops being about "did you remember the layering" and starts being about the actual design.

What code review becomes

Reviewers stop being layering police. They get to focus on naming, edge cases, and whether the abstraction is the right one.

Onboarding gets shorter

New engineers don't have to memorize invariants — they just see them fail. The codebase teaches itself.

Rolling it out without revolt

You don't need a flag day. Start by reporting violations as warnings. Once the team has cleaned up what's there, flip the same rule to error. Same code, harder enforcement.

Start with warnings, end with errors

This pattern works for every kind of architectural rule. Soft launch the constraint, give people time to react, then turn the dial up.

The takeaway

Architecture isn't a document. It's a constraint. Anything you can't express as a constraint will erode the day someone is in a hurry.

Make the rules executable, and the rules will hold.
